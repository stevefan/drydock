# Principal-protective friction — Forms A/B/C

**Status:** sketch · **Pulls from:** [vocabulary.md](vocabulary.md) §"Principal-protective friction", [harbor-authority.md](harbor-authority.md), [port-auditor.md](port-auditor.md)

## The premise

The V3 model has the principal at the top with full authority. Bucket 3 actions (destructive, expensive, irreversible) require principal-in-the-loop because the LLM-judgment loop isn't safe enough for them. But the principal is still human — tired, distracted, fat-fingered, occasionally operating on stale context. Sometimes the principal benefits from being slowed down.

This doc names the friction patterns that make principal-mistakes deliberate without making them impossible.

**The rule (load-bearing):** veto is never available. Friction makes the principal *look at* the action; it does not stop the principal *from doing* the action.

## The three forms

### Form A — Informational note

The lightest friction. The Auditor adds a one-line context to the principal's intended action when something looks worth noting:

```
🔎 about to destroy auction-crawl
   note: 50GB of unbacked state; you said yesterday you wanted to preserve it.
   proceed? reply 'yes' / 'no' / 'tell me more'
```

Properties:
- **Never blocking.** Principal replies 'yes' and the action runs.
- **Bypassable with one word.** No typing rituals.
- **Generated by Auditor LLM judgment** + standing rules (e.g., "always note unbacked-state on destroy"). Pure judgment notes default LIGHT and self-tune (notes that get 'yes'd 100% of the time should fire less often).
- **Audited.** The note + the response are recorded.

Use when: the action is reversible-but-worth-noting, OR irreversible-but-the-principal-probably-knows-but-might-not.

### Form B — Typed confirmation

Heavier. The principal must explicitly confirm by typing a token related to the action:

```
⚠️  destroy dock auction-crawl will:
    - delete container, worktree, secrets dir
    - keep volumes (per default destroy behavior)
type the dock name to confirm: _
```

Properties:
- **Declared in standing policy** (not LLM-judgment-derived). Principal pre-commits when calm: "any destroy command requires typed confirmation."
- **Bypassable per session.** `drydock incident-mode start --duration 30m` skips Form B confirmations until expiry. Use during real incidents where the friction would slow recovery.
- **The typed token is action-specific.** Dock name for destroy, bucket name for delete, lease ID for revoke. Reduces the chance a half-typed command runs.

Use when: the action is irreversible AND infrequent enough that the typing cost is worth the second-thought it produces. The AWS S3 bucket-deletion pattern.

### Form C — Mandatory cooldown

Heaviest. The action is queued for a delay; the principal can cancel within the window but cannot skip the delay:

```
⏱  cooldown started: destroy auction-crawl
   will execute in 60s
   reply 'cancel' to abort
   (cooldown skip requires policy update)
```

Properties:
- **Declared in standing policy** with explicit duration.
- **Cannot be skipped in the moment.** The only way to avoid the cooldown is to update the standing policy first (which has its own friction — at minimum a Form B confirmation that you're removing the cooldown).
- **The cooldown is the entire feature.** It's not "are you sure," it's "wait. think. decide."
- **The window is the cancel surface.** Reply 'cancel' to abort.

Use when: the action is so high-stakes that even a confirmed action benefits from a deliberate pause — production database deletion, mass credential rotation, tearing down a Yard. The "second-key" pattern from missile-launch systems and bank-wire-confirmations above $X.

## Picking which form for which action

The default mapping (principal can override per-action in policy):

| Action class | Default friction |
|---|---|
| Bucket-1 enforcement (deny capability, throttle on ceiling) | None — these are deterministic and the principal isn't in the loop |
| Bucket-2 Auditor actions (throttle, stop, freeze, revoke) | Form A — the Auditor explains why it acted, principal can override |
| Bucket-3 destroy a Dock | Form B (typed dock name) |
| Bucket-3 delete a volume | Form B + Form C (typed name + 30s cooldown) |
| Bucket-3 destroy a Yard | Form C (60s cooldown) + principal must endorse each member Dock individually |
| Bucket-3 provision new compute > $X | Form B (typed amount) + Form C (cooldown scaled by amount) |
| Bucket-3 modify a policy file | Form B (typed policy filename) |
| Bucket-3 rotate master credentials | Form C (long cooldown — 5 min); plus offer to defer to next business day |
| Routine ops (start, attach, exec, list, audit-read) | None |

Principal authors the standing policy at `~/.drydock/policy/friction.yaml`. Auditor enforces.

## What this is NOT

- **Not veto.** The Auditor never refuses to forward an action to Authority. Friction makes the principal's path more deliberate; it doesn't close the path.
- **Not paternalism.** Defaults are light. Routine actions have no friction. The Auditor's LLM-judgment notes (Form A) self-tune based on bypass rate.
- **Not LLM-discretionary blocking.** Form B and Form C are *declared* by the principal; the LLM doesn't decide to add them. Form A is LLM-discretionary but informational only.
- **Not a substitute for actual policy.** If the principal authors a policy that says "auction-crawl can't reach evil.com," the Authority enforces that deterministically — the Auditor isn't the gate. Friction is the layer ABOVE policy: the principal's own self-protection in the moment of high-stakes action.

## Anti-paternalism guards

These are baked-in design constraints, not just intentions:

1. **Defaults to NONE.** The system ships with no friction except on a small set of obviously-dangerous actions (destroy, delete, provision, rotate). Principal opts in to more.
2. **Self-tuning Form A.** Notes that the principal bypasses 100% of the time fire less often (down to never). The signal value of a note that's always ignored is zero; the cost is real.
3. **Per-session bypass.** `drydock incident-mode start` exists for a reason. Real incidents need fast iteration, not friction.
4. **No false confidence.** The Auditor is forbidden from saying "I checked, looks good." It can only say "here's what I noticed" or stay silent. Affirming the principal's intent makes the principal less attentive, which is the opposite of the goal.
5. **Audited bypass.** Frequency of bypass is itself a signal the principal can review. "Am I bypassing too much? Are the notes wrong too often?" — answerable from audit.
6. **No friction on the friction.** Updating the friction policy file itself does NOT carry friction (other than the Form B/C the principal explicitly chose to put on it). Otherwise the system traps you in your own settings.

## Implementation shape (sketch)

- **Policy file**: `~/.drydock/policy/friction.yaml` — principal-authored, daemon-readable.
- **Authority enforcement**: every Bucket-3 RPC checks the friction policy before executing. Returns either `friction_pending` (with the form-specific challenge) or proceeds.
- **CLI**: `drydock incident-mode start --duration 30m` enables per-session bypass for Form B (Form C cooldown is not bypassable mid-session — that's the point).
- **Telegram integration**: forms render in Telegram; principal can respond with the typed token / 'cancel' / 'yes'.
- **Audit shape**: every friction-bearing action emits a `friction.presented` event followed by `friction.bypassed` or `friction.cancelled` or `friction.satisfied` and the eventual `action.executed`.

## Resolved decisions and open questions

**Resolved:**
- **Form A self-tuning is transparent.** "I'm firing this note less often because you've bypassed it 30 times in a row" — surfaced to the principal so they can correct if needed.
- **Form C cooldown scaling.** Not for V3 single-principal; revisit when multi-user lands and very-high-stakes actions might warrant magnitude-scaled cooldowns or second-principal approval.
- **Friction boundary.** Applies at the *enforcement* boundary, not the *authoring* boundary. A direct YAML edit doesn't trigger friction; the policy applies when the change is *applied* via the daemon.
- **Per-scope friction overrides.** Yes — principal can declare friction overrides per Yard or per Dock (e.g., "ASI Yard requires Form C even for Bucket-2 actions").
